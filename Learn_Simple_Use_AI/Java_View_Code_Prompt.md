### 📐 核心排列规则

**三重排序原则：**

1. **类型依赖顺序**：被使用的类型必须先声明（内部类/接口按依赖关系逆序排列）
2. **成员变量顺序**：变量定义必须在使用的类型之后（若变量类型为内部类，则先定义内部类，再定义变量）
3. **方法调用逆序**：所有方法严格按照调用深度逆序排列（被调用的方法在前，调用者在后）尤其注意构造函数里调用其他方法

/** 最底层的类型（被其他类型使用，不依赖其他内部类型） */
public static class 基础类型 { ... }
public interface 基础接口 { ... }

/** 中间层类型（依赖基础类型） */
public static class 业务类型 { 
    private 基础类型 field;
}

/** 高层类型（依赖中间层类型） */
public static class 复杂类型 {
    private 业务类型 field;
}

// ========== 2. 成员变量（按变量类型依赖顺序）==========
/** 变量类型必须在变量定义前已声明 */

private 基础类型 基础变量;
protected 基础接口 接口变量;

// 使用业务类型的变量
private 业务类型 业务变量;
public static 业务类型 静态业务变量;

// 使用复杂类型的变量
private 复杂类型 复杂变量;
protected static final 复杂类型 常量变量;

// JDK原生类型变量（无依赖要求，可放在最后）
private String 字符串变量;
private int 整数变量;

// ========== 3. 方法定义（严格按调用链逆序）==========

/** ----- 第1层：基础方法（调用深度最低）----- */
// 不调用其他业务方法，只提供原子功能
private ResultSet 执行SQL查询() { ... }           // 被上层调用
private 用户 转换结果为用户() { ... }              // 被上层调用

// 组合基础方法，构建中间功能
private 用户 根据ID查询用户() {                     // 被上层调用
    执行SQL查询();                                 // 调用第1层
    转换结果为用户();                              // 调用第1层
}

/** ----- 第3层：业务方法（调用第2层）----- */
// 实现具体业务逻辑
private void 处理完成回调() {                       // 被入口调用
    根据ID查询用户();                              // 调用第2层
}

/** ----- 第4层：入口方法（调用第3层）----- */
// 对外暴露的API接口

}

// ========== 4. 构造函数（最后放置）==========
// 可能调用上述各种初始化方法
